# Summary Guidelines

## Detail Fields

Each paper entry includes three expanded detail fields, displayed
when the user clicks "자세히" on a paper card.

### `problem` (1-2 Korean sentences)

Describe the limitation or gap the paper addresses. Be specific:
mention concrete prior approaches, scenarios, or assumptions that
fall short. Avoid hype phrases like "critical issue" or "major
challenge" unless the paper itself uses measured language.

### `method` (1-2 Korean sentences)

Describe what the paper proposes. Mention the key technical
approach, framework component, or experimental design. Keep it
concrete: name the method, architecture, key loss, dataset, or
evaluation protocol. Avoid vague praise.

### `takeaway` (1-2 Korean sentences)

State the main result or implication. Include limitations when
they are evident from the abstract (e.g., "specific to X domain",
"evaluated only on synthetic data", "improvement over baseline
but still lags human performance").

## Tone

- Cautious and factual.
- No promotional language.
- When the approach has obvious limitations mentioned in the
  abstract (small evaluation set, narrow domain, synthetic data
  only), include that limitation in the takeaway.

## Source

These summaries are generated by `scripts/collect-papers.mjs`.
When `OPENCODE_GO_API_KEY` is available, the script uses
opencode-go/deepseek-v4-flash. Without a key, it falls back to
rule/heuristic templates. Review and improve when quality matters.

## Automation note

- Daily collection runs via GitHub Actions (`.github/workflows/daily-papers.yml`).
- Generated data goes to `data/papers.js`.
- No API keys or secrets are stored in the browser code.
- The static site works by opening `index.html` directly.
